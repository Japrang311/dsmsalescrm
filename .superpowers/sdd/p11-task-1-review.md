## Noted, not blocking

- **Minor — `public.current_user_role()` calls are not wrapped in `(select ...)`.** Every policy in this migration wraps `auth.uid()` as `(select auth.uid())` (the Postgres RLS performance pattern that lets the planner treat it as a single per-statement initplan instead of re-evaluating per row), e.g. `:80`, `:89`, `:98`. But `public.current_user_role()` calls throughout the same policies (`:81, 90, 99, 103, 116, 130...` and the `sales_orders_new`/`sales_order_items` equivalents) are left unwrapped as bare `public.current_user_role() in (...)`. Compare this against the *harden* migration this task was told to match the shape of (`20260718171152_harden_super_admin_rls_matrix.sql`), which consistently wraps every occurrence as `(select public.current_user_role())` (e.g. `:42-43, 52-53, 58-59`) specifically for this performance reason. This is not a correctness or security defect — RLS still enforces the same access rules either way — but it is a real, verifiable deviation from the established pattern the review brief asked to compare against, and if it isn't caught now it will likely get copy-pasted into more tables as Phase 11 proceeds. Recommend a small follow-up (in Task 2 or a quick fix-up migration) to wrap `public.current_user_role()` the same way, before more tables inherit the unwrapped form.
