## Goal

Make every RFQ, Quotation, and Sales Order line item identifiable by product; align the forms with the live Google Sheet headers; calculate weighted quotation forecast; and generate collision-safe Quotation/SO/NP/PROTY numbers after historical Sheet migration.

The user still creates each business record once in one form. Header and line-item separation is an internal database design, not a second user workflow.

## Authoritative Business Decisions

- One RFQ, Quotation, or Sales Order document can contain multiple line items.
- Every new line item requires `Nama Product`, `Qty`, and `UOM`.
- `Description` is optional. The Sales Order UI uses `Deskripsi Project` for the same field.
- UOM values are `Unit`, `Pcs`, `Set`, and `Lot`.
- Paid line total is always `Qty × Unit Price`; users cannot override it.
- Prototype FOC still records Product, Description, Qty, and UOM, but Unit Price and Total remain `NULL` and do not contribute to revenue.
- One `Create Sales Order` form records Nomor PO, the administrative SO number, and the ordered products. There is no second Customer PO transaction or separate PO-detail form.
- Revenue equals the sum of the recorded paid line items. The SO number is administrative and does not create, move, or change revenue.
- Revenue period is determined by the `Date` entered on Create SO.
- Date is displayed compactly as `18 Jul 2026` and stored as ISO `2026-07-18`.
- Existing rows are not given a fabricated Product Name. The UI displays `Nama Product belum diisi` until corrected manually.
- The list/detail UI groups items under one document header.

## Form Design

### RFQ

Header:

- Nomor RFQ: required; the existing non-official convenience default may remain.
- Date: required.
- Client and Sales: resolved from the selected client and owner.
- Stage: defaults to `Client Request for Quotes`.

Items:

- Nama Product: required, trimmed, maximum 200 characters.
- Description: optional, trimmed, maximum 500 characters.
- Qty: required, greater than zero.
- UOM: required dropdown.
- Unit Price: required, greater than zero.
- Total: calculated and read-only.

### Quotation

Header:

- Nomor Quotation: generated by the database after submit.
- Date: required; defaults to today.
- Account/Sales: resolved through owner.
- Client: resolved through client.
- Address: optional per-document value. It is not auto-filled because `public.clients` currently has no address column and historical values can represent a workshop/project site.
- Status: required weighted stage; defaults to `Quotes Sent`.
- SO Number: optional administrative link, normally filled after Closed Won.
- Note: optional.

Items are Nama Product, Description, Qty, UOM, Harga Satuan, and calculated Harga Total.

### Quotation revision

- `Buat Revisi` copies the latest version's header and all items into an editable form.
- The base number does not change.
- Canonical suffixes are `_REV.1`, `_REV.2`, and so on.
- Example: `DSM-26QUO-0405` → `DSM-26QUO-0405_REV.1` → `DSM-26QUO-0405_REV.2`.
- The previous version remains immutable history and is internally marked superseded.
- Only the latest revision contributes to quotation totals and forecast.
- A new revision defaults back to `Quotes Sent` because the revised quotation is sent again.
- Creating a revision and marking the previous version superseded occur in one database transaction.

### Sales Order

Header:

- Date: required, defaults to today, controls revenue Month/Year, displayed as `18 Jul 2026`.
- Nomor PO Customer: required.
- Nomor SO: assigned automatically for normal mode or entered manually for HARIFF backdate mode.
- Customer and Sales: resolved from client and owner.
- Existing SO type, PPN/Non-PPN, prototype status, and source fields remain.

Paid items:

- Nama Product, optional Deskripsi Project, Qty, UOM, Unit Price, calculated Total Price.

Prototype FOC items:

- Nama Product, optional Deskripsi Project, Qty, and UOM.
- Unit Price and Total Price remain `NULL`.

One submit creates one header and all items atomically. Revenue is the paid grand total of those items; assigning or correcting the administrative SO number does not modify revenue.

## HARIFF Numbering Mode

When the selected client is `PT. HARIFF DAYA TUNGGAL ENGINEERING`, show an explicit `Mode Penomoran SO`:

1. `SO Baru — Nomor Otomatis` (default)
   - Uses the normal SO/NP/PROTY series based on classification and the form Date's year.
   - Consumes the corresponding shared counter.
2. `SO Existing / Backdate — Nomor Manual`
   - Requires the official historical SO number and `Alasan Backdate`.
   - Does not consume any automatic counter.
   - Rejects a number already registered to another Sales Order header.
   - Writes an activity-log entry and displays a `Backdate` badge.

The backdated number is administrative only. Example: a HARIFF PO recorded on 18 Jul 2026 with manual number `DSM-22SO147` contributes revenue to July 2026, not 2022.

Historical HARIFF numbers such as `DSM-22SO122` through `DSM-22SO146` are preserved exactly during import. They are not continued automatically.

## Automatic Numbering Rules

Number allocation is per series and per two-digit year derived from the form Date:

| Document/classification | Format           | 2026 imported maximum observed | First candidate after import |
| ----------------------- | ---------------- | -----------------------------: | ---------------------------- |
| Quotation               | `DSM-YYQUO-0000` |                            404 | `DSM-26QUO-0405`             |
| SO PPN                  | `DSM-YYSO000`    |                            143 | `DSM-26SO144`                |
| SO Non-PPN              | `DSM-YYNP000`    |                             16 | `DSM-26NP017`                |
| SO Prototype            | `DSM-YYPROTY000` |                              8 | `DSM-26PROTY009`             |

Rules:

- QUO, SO, NP, and PROTY are independent counters.
- A new year starts a new counter at 1 for that series.
- Counter seeding uses the maximum unique valid base number imported for the series/year, not row count or physical last row.
- Duplicate line-item rows sharing one document number do not advance the counter.
- Gaps such as missing `NP014` are not filled; allocation continues from the maximum.
- Historical formats `_REV.01` and `_REV.1` are parsed to revision integer `1`; their raw imported number is preserved. Newly created revisions always use canonical `_REV.n`.
- Allocation and header insert occur transactionally in PostgreSQL so concurrent submissions cannot receive the same number.

## Weighted Stages and Forecast

| Stage stored on the commercial document | Weight |
| --------------------------------------- | -----: |
| `Client Request for Quotes`             |   0.15 |
| `Quotes Sent`                           |   0.30 |
| `Negotiation`                           |   0.55 |
| `Hot Prospect`                          |   0.75 |
| `Commit`                                |   0.90 |
| `Closed Won`                            |   1.00 |
| `Closed Lost`                           |   0.00 |

RFQ defaults to `Client Request for Quotes`; Quotation and new revisions default to `Quotes Sent`.

Forecast is derived, not stored separately:

`forecast = sum(current_version_line_total) × stage_weight`

Only the latest Quotation revision contributes. An unmapped historical stage displays `Belum tersedia` rather than silently receiving weight zero.

## Database Identity

- Local Supabase project ID: `DSM_SALES_WEB_APP_V2`.

- Public application schema: `public`.
- Internal non-Data-API schema for counters/functions: `private`.

## One-Form Transaction Boundary

The frontend does not insert a header and items through unrelated requests. A database transaction/RPC will:

1. Validate the authenticated actor and ownership scope.
2. Allocate or validate the document number.
3. Insert one header.
4. Insert all items.
5. Calculate the paid grand total or preserve FOC `NULL` values.
6. Write the corresponding activity-log entry.
7. Return the complete created document.

Any failure rolls back all steps; users never receive a header without its items or consume a counter without a created document.

## Existing Data and Foreign-Key Migration

The current database stores one `public.commercial_items`/`public.sales_orders` row per line item and repeats document headers. Migration will:

1. Group RFQ rows by `rfq_number`, Quotation rows by parsed Quotation version, and Sales Order rows by their existing SO number plus compatible header metadata.
2. Create one new header for each group and migrate every source row into its item table.
3. Preserve original UUID mappings in a migration map table during conversion.
4. Repoint `tasks.commercial_item_id`, `follow_up_logs.commercial_item_id`, and `activity_log.commercial_item_id` to their new commercial document header relationship.
5. Repoint `activity_log.sales_order_id` to the new Sales Order header.
6. Parse historical Quotation revisions without silently changing their raw number.
7. Leave Product Name `NULL` for historical rows that only have Description.
8. Preserve FOC value `NULL` and exclude it from revenue.
9. Move legacy source tables into the non-exposed `private` schema as read-only migration evidence until UAT and count reconciliation are approved. Their later deletion is a separate destructive decision.

Foreign-key column names will be clarified during migration:

- `tasks.commercial_document_id`
- `follow_up_logs.commercial_document_id`
- `activity_log.commercial_document_id`
- `activity_log.sales_order_id` remains but points to the header table.

## Sheet Import and Counter Seeding Order

1. Create the normalized local schema and importer support.
2. Import historical Sheet documents and line items.
3. Reconcile document counts, item counts, paid totals, FOC totals, and revision grouping.
4. Seed each `(series, year_code)` counter to the maximum imported base number.
5. Prove the next generated number is the expected candidate without leaving a test allocation behind.
6. Enable new-record creation only after reconciliation passes.

If the source Sheet changes before the real import, the importer recalculates maxima; it does not hardcode the observed July 18 values.

## Migration File Naming

Migration files are created with the Supabase CLI, never by inventing timestamps:

```sh
supabase migration new normalize_commercial_documents
supabase migration new migrate_commercial_document_data
supabase migration new add_atomic_document_numbering
```

Generated patterns:

- `supabase/migrations/<timestamp>_normalize_commercial_documents.sql`
- `supabase/migrations/<timestamp>_migrate_commercial_document_data.sql`
- `supabase/migrations/<timestamp>_add_atomic_document_numbering.sql`

The split keeps schema creation, data conversion, and allocator logic independently reviewable. Exact filenames are recorded after CLI generation.

## Security and RLS

- Every new `public` table has RLS enabled.
- Sales can create/read owned documents; managers can create/read across the permitted scope; executives remain read-only.
- Header and item policies use parent ownership consistently; `TO authenticated` alone is never treated as authorization.
- Counter tables/functions remain in `private` and are not directly callable through the Data API.
- Transaction functions use a locked `search_path`, explicit authorization checks, minimum grants, and no public default execute privilege.
- Existing policies are tested after FK migration.

## List and Detail UI

- RFQ groups by its header; Quotation groups by the exact version header; Sales Order groups by its header.
- Product Name is the primary item title. Empty Description is omitted.
- Detail pages show Product, Description, Qty, UOM, Unit Price, and Total.
- Prototype FOC hides money columns and shows `FOC — tidak masuk revenue`.
- Quotation history shows base and revisions in order, with the latest version marked active and older versions marked superseded.
- Commercial list cards show document total and weighted forecast only for the current revision.
- Sales Order lists show compact Date, Nomor PO, administrative Nomor SO, Customer, Sales, grand total, and backdate badge where applicable.

## Verification

1. Migration tests verify table names, headers, types, checks, indexes, RLS, grants, and FK destinations.
2. Historical reconciliation proves source row counts, grouped document counts, item counts, totals, FOC nulls, and activity/task relationships.
3. Concurrency tests submit simultaneous requests and prove unique monotonic numbers with no gaps caused by rolled-back inserts.
4. Counter tests cover yearly reset, independent series, imported maxima, skipped historical numbers, and HARIFF backdate non-consumption.
5. Revision tests cover copy, `_REV.n`, immutable history, current-version uniqueness, and forecast exclusion of superseded versions.
6. Form tests cover Product/UOM requirements, optional Description, paid price validation, FOC null money, and HARIFF modes.
7. Revenue tests prove the grand total comes from line items and is unaffected by SO-number assignment/correction.
8. Browser verification creates RFQ, Quotation, Quotation revision, PPN SO, NP SO, paid Prototype, FOC Prototype, and HARIFF backdate against local Supabase.
9. Browser evidence verifies compact Date `18 Jul 2026`, list/detail grouping, stage forecast, and exact persisted headers/items.
10. QA records are removed by captured IDs; baseline counts and counter state are restored without broad deletion.
11. Full tests, targeted lint, typecheck, production build, and database advisors are run and reported honestly.

## Acceptance Criteria

- Users record each RFQ, Quotation, or Sales Order once, with multiple line items in one form.
- Product, optional Description, Qty, UOM, and monetary behavior match the approved rules.
- Revenue equals the paid PO/order line-item total and uses the form Date; the SO number remains administrative.
- QUO/SO/NP/PROTY numbers are unique, sequential per series/year, atomically allocated, and seeded from imported maxima.
- Quotation revisions use canonical `_REV.n`; only the latest revision affects forecast.
- HARIFF supports normal automatic numbering and audited manual backdate numbering without changing the revenue period.
- Database names and every UI-to-column mapping are documented and inspectable.
- Existing history, audit links, and task/follow-up relationships survive normalization.
- No historical Product Name or stage is fabricated.
- No Google Sheet, production database, linked remote database, or published Git history is changed without separate authorization.

## Out of Scope

- A separate Customer PO entity or second PO-entry workflow.
- Writing back to the referenced Google Sheet.
- Automatic historical Product Name backfill.
- Silent historical-stage normalization.
- Deleting preserved legacy tables before separate approval.
- Applying migrations to a remote Supabase project.
